SQL
인젝션

SQL INJECTION — 정의

입력값을 검증하지 않아
SQL 문이 변조되는
취약점

사용자 입력이 쿼리의 일부로 직접 포함될 때, 공격자가 의도하지 않은 SQL 명령을 실행시킬 수 있습니다. 서버는 입력값과 쿼리를 구분하지 못해 악성 코드를 정상 쿼리로 처리합니다.

핵심 원인: 데이터(입력값)와 코드(SQL 명령)의 경계가 모호할 때 발생합니다.
피해 유형
  • 인증 우회 — 비밀번호 없이 로그인
  • 데이터 탈취 — 전체 테이블 조회
  • 데이터 변조 — UPDATE / DELETE 실행
  • 서버 명령 실행 — xp_cmdshell 등
  • 블라인드 인젝션 — 참/거짓으로 정보 유출
실제 사례

2011년 Sony PlayStation Network — SQL 인젝션으로 7,700만 계정 탈취.

① 취약한 코드

// 사용자 입력을 직접 쿼리에 삽입 ❌ $username = $_POST['username']; $password = $_POST['password']; $query = "SELECT * FROM users WHERE username = '$username' AND password = '$password'";

② 공격자의 입력값

-- username 필드에 입력: admin' OR '1'='1' -- -- 생성되는 최종 쿼리: SELECT * FROM users WHERE username = 'admin' OR '1'='1' --' AND password = '' -- '1'='1'은 항상 TRUE → 인증 우회!

③ 다양한 인젝션 패턴

-- Union 기반: 다른 테이블 조회 ' UNION SELECT username, password FROM admin_users -- -- 에러 기반: 오류 메시지로 정보 수집 ' AND EXTRACTVALUE(1, CONCAT(0x7e, (SELECT version()))) -- -- 블라인드: 참/거짓 응답으로 추론 ' AND SUBSTRING( (SELECT password FROM users LIMIT 1),1,1)='a -- Time-based: 응답 지연으로 추론 ' AND SLEEP(5) --
주석 처리 문자: MySQL — -- (공백 필요) 또는 #, Oracle — --, MSSQL — -- 또는 /* */ 을 이용해 쿼리 나머지를 무효화합니다.

SQL INJECTION — 주요 유형
01

In-band SQLi

공격과 결과 수집에 같은 채널을 사용. 가장 일반적인 유형으로 Union 기반과 Error 기반으로 나뉩니다.

02

Blind SQLi

오류 메시지가 노출되지 않아도 참/거짓 응답 차이나 시간 지연을 통해 데이터를 한 비트씩 추출합니다.

03

Out-of-band SQLi

DNS 요청이나 HTTP 요청을 통해 데이터를 외부로 전송. 방화벽 우회에 유리합니다. DNS exfiltration이 대표적입니다.

04

인증 우회

OR 1=1 구문으로 WHERE 조건을 항상 참으로 만들어 패스워드 없이 로그인합니다.

05

Stored SQLi

악성 페이로드를 DB에 저장해두고, 다른 사용자가 조회할 때 실행됩니다. XSS와 결합되는 경우가 많습니다.

06

OS Command 실행

MSSQL의 xp_cmdshell 등을 통해 DB를 넘어 운영체제 명령까지 실행 가능합니다.

SQL INJECTION — 방어법

✓ Prepared Statement

// 파라미터 바인딩으로 SQL 분리 $stmt = $pdo->prepare( "SELECT * FROM users WHERE username = ? AND password = ?" ); $stmt->execute([$username, $password]); // 입력값은 절대 SQL 코드로 해석 안 됨!
// Java — PreparedStatement String sql = "SELECT * FROM users WHERE username=? AND password=?"; PreparedStatement ps = conn.prepareStatement(sql); ps.setString(1, username); ps.setString(2, password);
다층 방어 전략
  • Prepared Statement — 입력값과 SQL 코드를 완전히 분리
  • Stored Procedure — DB에 사전 컴파일된 쿼리만 사용
  • ORM 사용 — Hibernate, SQLAlchemy 등 파라미터 자동 처리
  • 입력값 화이트리스트 — 허용 문자 패턴만 통과
  • 최소 권한 원칙 — DB 계정에 필요한 권한만 부여
  • WAF 적용 — 알려진 인젝션 패턴 차단
  • 오류 메시지 숨김 — DB 구조 노출 방지
핵심 원칙: 어떤 상황에서도 사용자 입력을 SQL 문자열에 직접 연결(concatenation)하지 마세요.

PART 02 — CROSS-SITE REQUEST FORGERY
CSRF

인증된 사용자의 브라우저를 이용해 해당 사용자 모르게 악의적인 요청을 전송하게 만드는 공격입니다. 서버는 요청이 정상 사용자에게서 온 것으로 판단합니다.

Cross-Site Request Forgery
07 / 14
CSRF — 정의

사용자의 신뢰된 세션을
무기로 사용하는
요청 위조 공격

브라우저는 동일 도메인으로의 요청에 쿠키를 자동으로 첨부합니다. 공격자는 이 특성을 악용해 피해자가 의도하지 않은 요청을 서버에 전송하게 만듭니다.

핵심 조건: 피해자가 대상 사이트에 로그인된 상태여야 하며, 서버가 요청 출처를 검증하지 않을 때 성립합니다.
공격 시나리오
  • 피해자가 banking.com에 로그인 — 세션 쿠키 보유
  • 공격자가 evil.com에 악성 페이지 준비
  • 피해자가 evil.com을 방문 (이메일, 링크 등)
  • evil.com이 banking.com으로 이체 요청 자동 전송
  • 브라우저가 banking.com 쿠키를 자동 첨부
  • 서버는 정상 요청으로 판단하고 이체 처리
피해 유형

계좌 이체, 비밀번호 변경, 이메일 변경, 상품 구매, 관리자 계정 생성, 설정 변경 등 — 피해자 권한으로 가능한 모든 동작.

피해 발생

공격자의 evil.com 페이지

<!-- 방법 1: 이미지 태그 (GET 요청) --> <img src="https://bank.com/transfer ?to=attacker&amount=1000000" style="display:none"> <!-- 방법 2: 자동 제출 폼 (POST 요청) --> <form action="https://bank.com/transfer" method="POST" id="hack"> <input name="to" value="attacker"> <input name="amount" value="1000000"> </form> <script> document.getElementById('hack').submit(); </script>

실제 전송되는 HTTP 요청

POST /transfer HTTP/1.1 Host: bank.com Origin: https://evil.com ← 출처 다름 Cookie: session=abc123xyz ← 자동 첨부! Content-Type: application/x-www-form-urlencoded to=attacker&amount=1000000 ------------------------------------------ 서버 입장: 유효한 세션 쿠키 → 정상 사용자 요청으로 판단 → 처리!
왜 작동하나? 브라우저는 도메인이 같으면 쿠키를 자동 첨부합니다. 서버가 요청의 출처(Origin)를 검증하지 않으면 구분이 불가능합니다.
09 / 14
CSRF — 공격 유형 & 특성
GET 기반 CSRF

이미지 태그, iframe, 링크 클릭을 통한 공격. 서버가 GET 요청으로 상태 변경을 허용할 때 발생.

<img src="https://target.com /logout"> <!-- 강제 로그아웃 -->
POST 기반 CSRF

숨겨진 폼 자동 제출. 계좌이체, 정보 변경 등 민감한 작업에 주로 사용됨.

SQL 인젝션 vs CSRF 비교
구분 SQL 인젝션 CSRF
공격 대상 서버/DB 사용자 브라우저
필요 조건 DB 쿼리 노출 사용자 로그인
주요 피해 데이터 탈취 권한 남용
탐지 난이도 중간 높음
중요: CSRF는 XSS(Cross-Site Scripting)와 다릅니다. XSS는 스크립트를 주입하지만, CSRF는 사용자의 기존 권한을 악용합니다.

CSRF — 방어법

① CSRF 토큰 방어 (핵심)

<!-- 서버가 랜덤 토큰 생성 --> <form method="POST" action="/transfer"> <input type="hidden" name="csrf_token" value="a3f9x2k7m1p..."> <!-- 공격자는 이 토큰값을 모름 --> </form>
// 서버에서 토큰 검증 if ($_POST['csrf_token'] !== $_SESSION['csrf_token']) { die('CSRF 공격 감지!'); } // 토큰 일치 시에만 처리 진행
다층 방어 전략
  • CSRF 토큰 — 요청마다 검증 가능한 랜덤 값 포함
  • SameSite 쿠키 — SameSite=Strict/Lax 설정으로 크로스사이트 쿠키 전송 차단
  • Origin/Referer 검증 — 요청 출처 확인 (우회 가능성 있어 보조 수단)
  • Double Submit Cookie — 쿠키와 폼 값을 동시에 검증
  • 재인증 요구 — 중요 동작 시 비밀번호 재확인
  • CORS 정책 — 허용된 Origin만 처리
// SameSite 쿠키 설정 예시 Set-Cookie: session=abc123; SameSite=Strict; HttpOnly; Secure

DEFENSE — 종합 보안 전략

단일 방어로는 부족합니다.
심층 방어(Defense in Depth)로
여러 계층을 중첩하세요.

01

입력 검증 (Input Validation)

모든 사용자 입력에 서버 측 화이트리스트 검증. 클라이언트 검증만으로는 부족합니다.

02

Parameterized Query

SQL 인젝션의 근본 해법. ORM 또는 Prepared Statement를 항상 사용하세요.

03

CSRF Token

모든 상태 변경 요청에 서버에서 생성한 랜덤 토큰을 포함하고 검증합니다.

04

보안 쿠키 설정

HttpOnly, Secure, SameSite=Strict 속성으로 쿠키를 강화합니다.

05

최소 권한 원칙

DB 계정, API 권한, 파일 접근 권한을 필요한 최소 수준으로 제한합니다.

06

WAF + 모니터링

Web Application Firewall로 알려진 공격 패턴을 차단하고 이상 트래픽을 감지합니다.

12 / 14
DEFENSE — 개발자 체크리스트

SQL 인젝션 방어

  • 모든 DB 쿼리에 Prepared Statement 또는 Parameterized Query 사용
  • ORM(Hibernate, SQLAlchemy, ActiveRecord) 활용
  • 저장 프로시저 사용 시에도 동적 SQL 지양
  • DB 사용자에게 SELECT만 필요한 경우 INSERT/UPDATE 권한 제거
  • 오류 메시지에 DB 구조 노출 금지
  • 정기적인 SQL 인젝션 취약점 스캔 (sqlmap, OWASP ZAP)
  • 코드 리뷰 시 문자열 연결 쿼리 패턴 집중 검토

CSRF 방어

  • 상태 변경 요청(POST/PUT/DELETE)에 CSRF 토큰 필수 포함
  • 프레임워크 내장 CSRF 보호 기능 활성화 (Django, Laravel 등)
  • SameSite=Strict 또는 Lax 쿠키 속성 설정
  • 중요 작업 수행 전 Origin/Referer 헤더 검증
  • 민감 작업(이체, 비밀번호 변경)에 재인증 요구
  • GET 요청으로 상태 변경 동작 구현 금지
  • 세션 만료 시간 적절히 설정
